# WissensNest — Knowledge Workbench Design

## Motivation

Conversations are ephemeral: messages pile up, context grows, and the useful parts are buried in
noise. The Knowledge Workbench adds a curated authoring layer alongside the chat — a place to
capture what matters, organise it, and export it.

The mental model is a research workflow:

1. Explore a topic in a conversation.
2. Promote the best AI responses to blocks in an article.
3. Edit, reorder, merge, and annotate the blocks until the article says exactly what you want.
4. Export selected blocks as a PDF.

---

## Entity Hierarchy

```text
Project
  ├── Section
  │     ├── Conversation  (SectionId nullable FK — conversations can live inside a section)
  │     └── Article
  │           └── Block   (atomic Markdown chunk; same data shape as Message)
  └── Conversation        (conversations without a section belong directly to the project)
```

`SectionId` on `Conversation` is a nullable FK with `SetNull` on delete — if a section is
deleted, its conversations move back to the project level rather than being deleted.

---

## Block as the Atomic Unit

A `Block` is a self-contained piece of Markdown stored as a plain string with an `Ordinal`
(integer, controls display order) and an optional `SourceMessageId` back-link to the `Message`
it was promoted from.

Blocks are the unit of:

- **Editing** — each block has its own edit/preview toggle.
- **Reordering** — ↑ / ↓ swaps ordinals with the adjacent block.
- **Moving and copying** — a block can be moved or copied to any article in the same project.
- **Merging** — two adjacent blocks join their content with a blank line; the second is soft-deleted.
- **Splitting** — at a cursor offset; the tail becomes a new block inserted below.
- **PDF export** — whole article or a user-selected subset.

## Conversation ↔ Article Bridge

![Knowledge Workbench bridge flow](../../Images/23_02_WissensNest_KnowledgeWorkbench_BridgeFlow.svg)

### Promote Message → Block

Any persisted message has a **→ Block** button in its hover toolbar. Clicking it opens a lazy-loaded section/article picker. On confirmation, `POST /articles/{id}/blocks` is called with `sourceMessageId` set to the message's ID. The block lands at the end of the target article.

### Send Block → Conversation

Any block in the Article Editor has a **→ Chat** button. Clicking it sets `ChatState.PendingSeedContent` and navigates to `/chat`. `Chat.razor` detects the seed on `OnChatStateChanged`, pre-fills the input, and clears the seed. The block content becomes the opening message of a new conversation.

---

## Scratch Area

Each section has exactly one **Scratch** article — a catch-all inbox for quick capture.
`GET /sections/{id}/scratch` finds or creates it. The sidebar shows a **📋 Scratch** entry
at the top of each expanded section. The Scratch article behaves like any other article
and its blocks can be moved to proper articles once organised.

---

## Advanced Block Operations

| Operation | API | How it works |
| --- | --- | --- |
| Move to article | `PATCH /blocks/{id}/move` | Block appended to end of target; disappears from source |
| Copy to article | `POST /blocks/{id}/copy` | Copy appended to end of target; original stays |
| Merge with next | `POST /blocks/merge` | Body `{firstId, secondId}` — content joined with blank line; second soft-deleted |
| Split at cursor | `POST /blocks/{id}/split` | Body `{splitAtOffset}` — tail becomes new block below; JS `getTextareaCursorPosition` reads `selectionStart` |

## PDF Export

`POST /articles/{id}/export` — body `{ format: "pdf", blockIds?: Guid[] }`.

Without `blockIds`, all blocks in ordinal order are exported. With `blockIds`, only the
selected blocks (in their article order) are included. The PDF is generated by `PdfExporter`
using **QuestPDF** (MIT licence, pure C#, no native dependencies) with **Markdig** for
Markdown-to-plain-text conversion. The API response is a binary `application/pdf` stream;
the UI triggers the browser download via a base64 JS interop call (`window.downloadFile`).
