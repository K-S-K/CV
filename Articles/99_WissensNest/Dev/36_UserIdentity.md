# WissensNest

## User Identity — Design Notes

This article covers the design of the lightweight identity system introduced to support family access over the LAN. The goal was the minimum infrastructure required to assign a stable, cross-device GUID to each family member — without passwords, accounts, or any external auth service.

---

### Problem Statement

With LAN access enabled (see `20_FamilyAccess.md`), all family members share the same Blazor Server application, the same database, and the same conversation history. Before introducing identity, the system had no way to distinguish who was talking. This blocks any future scope or ownership feature: you cannot give someone their "own" conversations or projects if the system does not know who they are.

The requirements were:

- No passwords. A family tool with 3–5 named users does not need them.
- Stable across browser sessions on the same device (survives reload, tab close).
- Portable across devices: if Mama opens the app on the Android phone for the first time, she should be able to claim the same identity she already has on the Windows laptop.
- Zero friction: no email, no verification, no sign-up form beyond entering a name.

---

### Identity Layers

Identity is maintained in two cooperating layers.

**Layer 1 — Browser (localStorage)**

Two keys are stored per browser:

| Key | Value |
| --- | --- |
| `wn_user_id` | UUID (GUID as lowercase string) |
| `wn_user_name` | Display name chosen by the user |

`wn_user_id` is generated by `crypto.randomUUID()` on the first visit if it does not already exist. The JS helper `window.wnIdentity.getUserId()` always returns a non-empty string — it creates the GUID if missing. This means every browser has a locally stable identity before it ever talks to the server.

**Layer 2 — Server (Users table)**

A `Users` table in the SQLite database holds the canonical registry:

| Column | Type | Notes |
| --- | --- | --- |
| `Id` | GUID | Matches the `wn_user_id` from the browser that registered |
| `Name` | TEXT | Human-readable display name, max 100 chars |
| `CreatedAt` / `UpdatedAt` | DateTimeOffset | Standard audit fields |
| `IsDeleted` / `DeletedAt` | bool / DateTimeOffset? | Soft-delete pattern |

The server record is created the first time a user enters their name and clicks Save. The GUID generated on the client is sent to `POST /users` as `id`, so the server record carries the same GUID already stored in `localStorage`. From that point on, localStorage and the DB are in sync.

---

### First-Visit Flow

```
Browser opens app
  │
  ├─ wn_user_id in localStorage?
  │   └─ YES → wn_user_name in localStorage?
  │               └─ YES → Identity confirmed, show badge, done
  │               └─ NO  → (shouldn't happen; show name input)
  │
  └─ NO → fetch GET /users
              │
              ├─ No users yet → show name input → RegisterAsync → done
              │
              └─ Users exist → show "Who are you?" picker
                                  │
                                  ├─ Pick existing → ClaimAsync (localStorage only) → done
                                  │
                                  └─ "I'm new" → show name input → RegisterAsync → done
```

`RegisterAsync` calls `POST /users` with the candidate GUID and the entered name, then writes both `wn_user_id` and `wn_user_name` to localStorage.

`ClaimAsync` is purely local: it writes the chosen user's GUID and name into localStorage without touching the server. The server already has the record from when that user first registered on their own device.

### Component Design

**`window.wnIdentity`** (in `wwwroot/js/interop.js`)

Four functions — `getUserId`, `setUserId`, `getUserName`, `setUserName` — over `localStorage`. `getUserId` auto-creates the GUID on first call. All are called via `IJSRuntime` from C#.

**`UserIdentityService`** (scoped, `WissensNest.UI/Services`)

One instance per Blazor circuit. Holds `UserId` and `UserName` in memory after `InitializeAsync()` so the rest of the circuit never needs to call JS again.

Public operations:

| Method | When used |
| --- | --- |
| `InitializeAsync()` | Called once from `NavMenu.OnAfterRenderAsync(firstRender)` |
| `GetKnownUsersAsync()` | Called by the modal to populate the picker list |
| `ClaimAsync(UserInfo)` | User picks an existing identity from the list |
| `RegisterAsync(name, proposedId)` | User enters a new name; creates server record |
| `RenameAsync(name)` | User changes their display name via badge click |

`NeedsIdentity` is `true` when `IsInitialized && string.IsNullOrEmpty(UserId)`. The nav menu watches this flag to decide whether to open the modal.

**`NavMenu.razor`** (`@rendermode InteractiveServer`)

Three-state modal controlled by `_loading`, `_showPicker`, and the default (name input) state:

1. **Loading** — shown briefly while `GET /users` is in flight.
2. **Picker** — shown when at least one user exists; lists names as clickable buttons plus an "I'm new" button.
3. **Name input** — shown when no users exist, or when the user clicks "I'm new", or when the badge is clicked to rename.

The modal cannot be dismissed without completing the flow on a first visit (no Cancel button). When renaming, Cancel is available and returns the modal to its closed state.

---

### API Endpoints

| Method | Path | Body | Response |
| --- | --- | --- | --- |
| `GET` | `/users` | — | `UserInfo[]` ordered by `CreatedAt` |
| `POST` | `/users` | `{ name, id? }` | `UserInfo` |
| `PATCH` | `/users/{id}` | `{ name }` | `UserInfo` |

`POST /users` accepts an optional `id` field. When present, the server uses that GUID for the new record rather than generating a new one. This is how the client-generated GUID is anchored to the server record.

---

### Dependency Chain

```
NavMenu.razor
  └─ UserIdentityService  (IJSRuntime, IWissensNestClient)
       └─ MyAiClient       (GET /users, POST /users, PATCH /users/{id})
            └─ WissensNest.API  (IUserRepository)
                 └─ UserRepository  (WissensNestDbContext → Users table)
```

`UserIdentityService` lives in `WissensNest.UI` and depends on `IWissensNestClient` (from `WissensNest.Contracts`). This means the UI project has no direct dependency on persistence — consistent with the existing layering.

---

### What Is Not Yet Done

The identity system currently does the minimum: every browser gets a GUID, and there is a server registry that names can be looked up from. Nothing in the application yet uses the GUID for filtering, ownership, or access control.

The next planned layer is **scopes** — associating Projects, Conversations, and Articles with a user ID so that each person can have their own space alongside shared common content. See `37_Scopes.md` (planned) for the design.

```bash
dotnet ef migrations add AddUsers \
  --project Src/Foundation/WissensNest.Persistent.SQLite \
  --startup-project Src/Foundation/WissensNest.Persistent.SQLite
```
